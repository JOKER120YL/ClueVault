# ClueVault PRD

## 1. 文档信息

- 产品名称：ClueVault
- 形态：Windows 桌面客户端
- 当前版本目标：v1
- 主要用户：运营、客服、测试协同人员
- 相关团队：运营团队、研发团队、测试组

## 2. 背景与目标

当前运营或客服在收集用户问题时，通常会拿到模型文件、截图、零散文字说明等材料。这些材料需要再整理后交给研发测试组，过程存在以下问题：

- 材料分散，模型、图片、备注经常不在一个地方
- 命名不统一，测试组很难快速判断问题内容
- 问题描述依赖人工整理，效率低，质量波动大
- 不同运营提交习惯不同，给研发测试带来额外沟通成本

本产品目标是提供一个 Windows 端统一工具，帮助运营快速完成问题收集、整理、总结和提交，最终把一条问题沉淀为一个标准化文件夹，直接进入研发测试流程。

## 3. 核心价值

- 降低客服和运营整理 bug 材料的时间成本
- 统一提交结构，让研发测试组更容易接手
- 借助大模型先生成可编辑草稿，提高问题摘要质量
- 支持多人安装使用，每位运营保留自己的本地身份与 API 配置

## 4. 用户角色

### 4.1 运营 / 客服

- 收到用户反馈后，整理模型、截图和说明
- 通过客户端提交问题包
- 必要时手动修正 AI 草稿

### 4.2 研发测试组

- 从共享目录读取标准化问题文件夹
- 根据 `说明.md` 和附件判断问题、安排排查或复现

### 4.3 管理者 / 实施同事

- 分发安装包给其他运营
- 提供共享目录地址规范
- 维护推荐的模型供应商配置说明

## 5. 使用场景

### 场景一：标准 bug 提交

运营收到一个模型和两张截图，知道大概现象但不想手动写长说明。软件协助生成标题、摘要和关注点，运营确认后直接提交到共享目录。

### 场景二：AI 不可用时的兜底提交

运营没有配置 API Key，或者当前模型接口失败。软件仍然生成本地草稿，保证提交流程不中断。

### 场景三：多人共用同一流程

多位运营各自在自己的电脑上安装软件。首次启动填写姓名/昵称并保存，后续默认以此身份提交，便于测试组追溯来源。

## 6. 产品范围

## 6.1 v1 包含

- Windows 桌面端应用
- 首次启动引导
- 本地保存姓名/昵称
- 本地保存共享目录与大模型配置
- 支持模型、图片、其他附件上传
- 支持填写问题描述和补充说明
- 支持 AI 草稿生成
- 支持用户人工修改草稿
- 支持创建标准化 bug 文件夹
- 支持生成 `说明.md`
- 支持 Windows 安装包打包配置

## 6.2 v1 不包含

- 登录系统
- 权限管理
- 云端提交历史
- 对接 Jira、禅道、飞书多维表等平台
- 自动识别模型内部几何或 BIM 属性
- 自动生成复现步骤
- 公司统一 Key 管理后台

## 7. 功能需求

### 7.1 首次启动引导

用户首次打开软件时，需要先完成初始化配置。

必填信息：

- 姓名 / 昵称
- 共享目录

可配置 AI 信息：

- 模型供应商
- Base URL
- 模型名
- API Key

要求：

- 未填写姓名 / 昵称时，不能视为初始化完成
- 配置保存到本地
- 后续打开软件时自动读取

### 7.2 个人身份管理

- 首次启动填写的姓名 / 昵称作为默认提交人
- 新建问题单时自动带入
- 在设置区允许修改默认值
- 在单次提交页允许临时改动
- 临时改动只对当前提交生效

### 7.3 共享目录配置

- 软件默认提供一个共享目录占位值
- 正式使用时建议填写真实可访问的 `UNC` 路径
- 软件支持校验目录是否存在且是否可写
- 校验失败时给出明确错误
- 提交前再次检查，避免半成功提交

### 7.4 问题单创建

每条问题单至少包含：

- 提交人
- 问题描述
- 附件列表

可选信息：

- 项目名
- 补充说明

要求：

- 问题描述是核心必填项
- 界面支持上传模型文件、图片、其他附件
- 附件允许多选
- 附件列表可移除

### 7.5 AI 草稿生成

软件支持通过 OpenAI 兼容接口生成草稿。

输入内容包括：

- 提交人
- 项目名
- 问题描述
- 补充说明
- 附件名称与类型

输出内容包括：

- 问题标题
- 问题摘要
- AI 分析结论

要求：

- 默认支持 `DeepSeek`
- 支持自定义 OpenAI 兼容接口
- AI 只生成草稿，不直接自动提交
- 用户必须可以手动修改标题、摘要、分析结论
- AI 调用失败时，自动降级为本地草稿生成

### 7.6 标准化提交

用户确认后，软件将问题落到共享目录。

目录规则：

- 每个 bug 创建一个独立文件夹
- 命名规则为 `YYYY-MM-DD_问题标题`
- 如果填写了项目名，则使用 `YYYY-MM-DD_项目名_问题标题`
- 文件夹名需清洗非法字符

文件夹内容：

- 原始模型文件
- 原始图片文件
- 其他补充附件
- `说明.md`

### 7.7 说明文档生成

`说明.md` 至少包含：

- 问题标题
- 提交人
- 创建时间
- 项目名（如有）
- 问题摘要
- 原始问题描述
- AI 分析结论
- 附件清单

### 7.8 结果反馈

提交成功后，界面应反馈：

- 最终目录名
- 写入位置
- 提交人
- 附件数量
- 文档是否生成成功

提交失败后，界面应反馈明确原因，例如：

- 共享目录不存在
- 无写入权限
- 目录重名
- 文件复制失败

### 7.9 悬浮窗助手

- 软件启动后默认显示一个桌面悬浮窗
- 悬浮窗支持点击打开主窗口
- 悬浮窗支持直接拖入文件
- 拖入文件后自动唤起主窗口，并把附件带入当前问题单
- 悬浮窗支持在设置中开启 / 关闭
- 悬浮窗支持根据“当日提交次数阈值”切换角色形象

角色规则：

- 默认角色为蛙弟，情绪为懵懂、呆萌、待命中
- 当天提交次数达到阈值后，切换为蜂哥，情绪为生气、无语、督促感更强
- 阈值默认 1，支持在设置中调整

## 8. 非功能需求

### 8.1 平台

- 支持 Windows 桌面环境
- 面向非技术运营同事使用

### 8.2 易用性

- 首屏要明确“先配置，再提交”
- 必填项清晰标识
- 提交流程尽量线性，不增加复杂学习成本
- 悬浮窗应适合小尺寸显示，状态切换需一眼可辨

### 8.3 稳定性

- AI 不可用时不阻断主流程
- 路径校验失败时不允许进入半提交状态
- 附件重复选择时应去重

### 8.4 安全性

- API Key 仅保存在本地配置中
- 不把 API Key 写入 `说明.md`
- 不向共享目录写入无关本地配置

## 10. 业务规则

- 姓名 / 昵称首次启动必填
- 问题描述为问题单核心必填项
- 提交时如果提交人为空，则优先回退到本地默认身份
- 提交目录必须可访问且可写
- 文件夹命名必须剔除非法字符
- AI 输出仅作辅助，不代表最终事实判断

## 12. 验收标准

### 12.1 首次启动

- 未填写姓名 / 昵称时，不能完成初始化
- 已填写后，重启软件仍能自动带出

### 12.2 标准提交流程

- 上传 1 个模型、若干图片和文字说明后，可成功生成文件夹
- 文件夹中存在全部附件与 `说明.md`
- `说明.md` 内包含提交人、摘要、描述和分析结论

### 12.3 AI 能力

- `DeepSeek` 预设可正确带出默认 Base URL
- 使用兼容接口时可正常生成草稿
- 接口失败时能回退到本地草稿

### 12.4 异常场景

- 共享目录不存在时，禁止提交并提示错误
- 无写入权限时，禁止提交并提示错误
- 问题标题中包含非法字符时，仍能生成合法目录名
