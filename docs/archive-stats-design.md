# 归档统计设计

## 目标

统计团队每天成功归档了多少次模型材料。

统计口径：

- 一次成功归档算 `1` 次。
- 一个归档文件夹中包含多个文件，也只算 `1` 次。
- 只有归档成功写入目标目录后才记录统计。

## 当前实现

每次归档成功后，软件会写两份统计事件：

- 本机明细：`%LOCALAPPDATA%\ClueVaultDesktop\archive-events.jsonl`
- 共享盘明细：`共享目录\_ClueVaultStats\YYYY-MM\archive-events-客户端ID.jsonl`

共享盘统计写入失败不会影响主归档流程，只会写入本地日志。

共享盘统计按月份和客户端拆分文件，避免多名运营同时归档时都追加写入同一个 JSONL 文件。
统计看板读取时会汇总共享盘内所有统计文件，并按 `EventId` 去重。
旧版 `共享目录\_ClueVaultStats\archive-events-YYYY-MM.jsonl` 文件仍会被读取，历史数据不会丢失。

## 事件字段

每一行是一条 JSON，表示一次归档事件：

```json
{
  "EventId": "归档唯一ID",
  "CreatedAt": "归档时间",
  "Date": "归档日期",
  "Submitter": "提交人",
  "MachineName": "电脑名",
  "ClientId": "客户端ID",
  "Source": "来源",
  "DisciplineId": "专业ID",
  "DisciplineLabel": "专业",
  "FolderName": "归档文件夹名",
  "TargetDirectory": "归档目录",
  "AttachmentCount": 2,
  "AppVersion": "软件版本"
}
```

`EventId` 可用于导入飞书表格时去重。

## 文件体积

JSONL 一行通常不到 1KB。

粗略估算：

- 每天 100 次归档：约 100KB/天。
- 每月 3000 次归档：约 3MB/月。

因此先按月份 + 客户端拆分共享盘统计文件即可，不需要数据库。

## 后续同步到飞书

后续可以由额外脚本或 Codex 读取共享盘中的 JSONL 文件，再通过 `lark-cli` 写入飞书表格。

建议飞书表格一行对应一次归档事件，而不是只存每天总数。这样后续可以按日期、人员、专业、来源灵活汇总。

导入时用 `EventId` 判断是否已存在，避免重复上传。

## 与经验库的关系

统计文件验证了“本地一份 + 共享盘一份”的协作模式。

未来经验库也可以采用类似方式：

- 本机保留自己的经验数据，避免共享盘异常导致数据丢失。
- 共享目录保存团队可见数据，便于其他人读取和同步。
- 每条记录保留唯一 ID，便于导入、导出和去重。
